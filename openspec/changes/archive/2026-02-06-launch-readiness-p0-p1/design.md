## Context

目标是把现有“可用”的协作清单提升到“可上线可持续使用”。现状包含私密 token 机制与多处 UI/文档不一致，导致加入流程断裂、错误反馈分散、生命周期不透明。变更涉及前端体验、协议文档与运行说明的同步对齐，不涉及核心数据模型或新基础设施。

## Goals / Non-Goals

**Goals:**
统一私密邀请心智，明确 token 为唯一入场凭证  
补齐连接状态、错误状态与生命周期提示  
暴露邀请安全控制入口并对齐文档  
提升权限可见性与回访失败提示  
对齐协议文档与现有实现

**Non-Goals:**
新增账号体系或复杂权限  
引入新的实时同步协议或存储方案  
完整操作历史与回滚

## Decisions

统一“邀请码/链接”作为加入入口  
理由：与私密邀请定位一致，避免 roomId 直入的误解  
替代方案：允许 roomId 直入或只读模式，因私密定位不匹配而拒绝

以 UI 统一状态提示替代散落的 alert  
理由：提升体验一致性，方便 QA 验收  
替代方案：保留 alert，仅修补文案，仍会导致状态割裂

文档以实现为准，集中更新协议与错误行为  
理由：确保未来多端接入或协作开发的一致性  
替代方案：仅局部修补 README，覆盖不足

邀请重置作为 Admin 显式入口  
理由：私密分享需要可控撤销  
替代方案：仅保留 API，无 UI 入口，用户无法自救

## Risks / Trade-offs

“邀请码/链接”心智会提高加入门槛 → 用更清晰的文案与复制入口降低摩擦  
错误状态集中化可能需要调整现有交互 → 通过一致的状态组件减少 UI 抖动  
生命周期提示可能影响体验（看起来“会过期”） → 在提示文案中强调“无操作才过期”
